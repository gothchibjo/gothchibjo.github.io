# Follow-Us: постановка задачи и требования

## 1. Цель
Сервис фиксирует итоги встречи в стандартизированном формате follow-up, сохраняет документ и позволяет быстро делиться им ссылкой и экспортом.

## 2. Пользовательские роли
- Автор follow-up: создает, редактирует, экспортирует документ.
- Участник встречи: открывает ссылку и читает итоговый документ.

## 3. Функциональные требования (MVP)
1. Создание follow-up документа с полями:
- дата встречи;
- заголовок;
- список участников ("Были");
- список обсужденных тем ("Обсудили");
- список решений ("Решили");
- список задач ("Поставили задачи": формулировка, ответственный, срок);
- служебные поля (например, хэштеги).

2. Формирование текста документа в едином шаблоне, совместимом с примером:
- секции с нумерацией;
- списки;
- компактный формат для пересылки.

3. Сохранение документа в сервисе:
- документ получает уникальный id;
- доступен список сохраненных документов;
- возможно повторное открытие и редактирование.

4. Доступ по ссылке:
- генерируется ссылка вида `.../#/doc/<id>`;
- по открытию ссылки документ загружается в форму.

5. Экспорт:
- выгрузка в `md`;
- выгрузка в `docx`;
- выгрузка в `pdf`.

## 4. Нефункциональные требования (MVP)
1. UX:
- ввод основных данных не дольше 2-3 минут;
- интерфейс адаптивен для desktop/mobile;
- предпросмотр документа обновляется при вводе.

2. Производительность:
- открытие интерфейса до 2 секунд на типовом ноутбуке;
- формирование предпросмотра в реальном времени без заметных задержек.

3. Надежность данных:
- черновики/документы не теряются при перезагрузке страницы (локальное хранилище в MVP).

4. Совместимость:
- актуальные версии Chrome/Edge/Safari/Firefox.

5. Безопасность (базовый уровень MVP):
- данные хранятся локально в браузере пользователя;
- без авторизации и без серверной публикации в первой версии.

## 5. Ограничения MVP
- ссылку можно открыть только в браузере, где документ сохранен (локальное хранение);
- нет коллективного редактирования;
- нет серверной авторизации и прав доступа;
- нет отчетов и сквозной аналитики.

## 6. Бэклог после MVP
- серверное хранение и публичные ссылки `followup.com/user/hash_link`;
- кросс-ссылки между follow-up документами;
- автоподстановка часто используемых участников/тем;
- отметки повторно обсуждаемых тем и просроченных задач;
- отчеты: список решений, список задач по срокам и ответственным.

## 7. Критерии приемки MVP
1. Пользователь может создать документ со всеми обязательными секциями.
2. Документ сохраняется и появляется в списке сохраненных.
3. Документ можно открыть из списка и по hash-ссылке.
4. Экспорт в `md`, `docx`, `pdf` создает скачиваемые файлы.
5. Шаблон итогового текста соответствует заявленной структуре.
